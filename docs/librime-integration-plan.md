# librime Integration Plan

This MVP does not vendor librime or any GPL input-method frontend code.

## Native dependencies to evaluate

- librime: BSD-3-Clause.
- Boost: Boost Software License 1.0.
- yaml-cpp: MIT.
- LevelDB: BSD-3-Clause.
- OpenCC: Apache-2.0.
- marisa-trie: BSD/LGPL dual licensing depending on package; prefer BSD-compatible distribution and verify before vendoring.
- glog: BSD-3-Clause.
- gtest: BSD-3-Clause, build-time tests only.

## OHOS NDK/CMake approach

1. Keep `native/rime_bridge` as the ArkTS-facing NAPI boundary.
2. Build third-party libraries as static libraries with the OHOS toolchain file generated by DevEco/SDK.
3. Link `rime_bridge` to `librime` only after every dependency builds for `arm64-v8a`.
4. Keep ArkTS API stable: `init`, `processKey`, `getComposition`, `getCandidates`, `commitCandidate`, `clearComposition`, `deploy`, `getStatus`.

## Current real-rime switch

The native bridge now has a real librime wrapper in `native/rime_bridge/rime_engine_real.cpp`.
Local OHOS `arm64-v8a` dependency artifacts can be built with:

```sh
native/third_party/build_ohos_deps.sh
```

The `entry` module currently requests the real bridge with:

```text
-DRIME_BRIDGE_USE_REAL_RIME=ON
```

`native/rime_bridge/CMakeLists.txt` auto-detects dependencies under `native/third_party/ohos/<abi>`. `arm64-v8a` links real `librime.a`; ABIs without matching artifacts fall back to the stub bridge.

Do not link Homebrew/macOS `librime.dylib` into HarmonyOS. It is the wrong ABI.

The ArkTS wrapper in `entry/src/main/ets/native/RimeNativeBridge.ets` now attempts to load `librime_bridge.so` first and falls back only if the native module is unavailable.

Rime runtime data paths are now centralized in `KeyboardController`:

```text
context.filesDir/rime/user
context.filesDir/rime/shared
```

The next implementation step is copying packaged schema/dict files into `shared` on first run, then invoking `deploy()`.

## Risk areas

- Boost and transitive compile flags are usually the largest build-system surface.
- OpenCC dictionary/data packaging needs careful resource placement.
- LevelDB file locking and path behavior must be verified inside the Harmony sandbox.
- marisa package licensing must be verified before inclusion.

## MVP simplifications

- Start with one schema, preferably `luna_pinyin`, and keep schema assets in app resources or copied sandbox storage.
- Disable optional plugins that require dependencies not yet ported.
- Keep user data under app sandbox, for example `context.filesDir + '/rime/user'`.
- Keep shared schema data under a read-only copied directory, for example `context.filesDir + '/rime/shared'`, or package resources copied on first run.
